1 Aislamiento con HERMES_HOME

Cada proyecto Hermes debe vivir aislado de tu ~/.hermes personal.

Hermes por defecto usa ~/.hermes como directorio raiz. Si tienes multiples proyectos (traduccion, chatbot, investigacion), todos comparten el mismo espacio y se contaminan.

La solucion: HERMES_HOME por proyecto

# En tu Makefile
export HERMES_HOME := $(CURDIR)/.hermes

# Todo lo que Hermes haga vive dentro del proyecto
# Tu ~/.hermes personal queda intacto
Regla: Siempre define HERMES_HOME en el Makefile del proyecto. Nunca dejes que un proyecto toque ~/.hermes. Agrega .hermes/ al .gitignore.

Que va en .hermes/ (runtime, gitignored)

ContenidoDescripcion
.envAPI keys (ANTHROPIC_API_KEY, GLM_API_KEY)
profiles/*/Copias instaladas de los profiles
state.dbBase de datos SQLite de memoria
sessions/Estado de sesiones activas
logs/Logs de runtime
Gotcha (macOS): Si tu directorio tiene espacios en el nombre (ej: Interpretation AI Cell), TODOS los paths en el Makefile que pases a sed u otros comandos deben ir entre comillas dobles: "$(CURDIR)/profiles/..."

2 Anatomia de un Profile

Cada agente es un profile con estructura estandarizada.

profiles/mi-agente/
  SOUL.md                # Identidad (slot #1 en system prompt)
  config.yaml            # Modelo, provider, kanban settings
  distribution.yaml      # Manifiesto para hermes profile install
  skills/
    skill-name/
      SKILL.md           # Procedimiento con frontmatter YAML
      (archivos auxiliares: registries, lookup tables...)
  cron/
    job-name.yml         # Tareas programadas
  memories/              # [gitignored] Memoria episodica
  sessions/              # [gitignored] Estado de sesion
  state.db               # [gitignored] SQLite FTS5

config.yaml minimo

model:
  default: claude-sonnet-4-20250514
  provider: anthropic

kanban:
  dispatch_in_gateway: true
  dispatch_interval_seconds: 5

distribution.yaml minimo

name: mi-agente
version: 1.0.0
description: "Descripcion corta del agente"
hermes_requires: ">=0.12.0"
author: "Tu Nombre"
license: "MIT"
env_requires:
  - name: ANTHROPIC_API_KEY
    required: false
Regla: distribution.yaml es obligatorio para que hermes profile install funcione. Sin el, Hermes no sabe como instalar tu profile.

3 SOUL.md — Identidad del Agente

La constitucion del agente. Se inyecta como slot #1 en cada prompt.

El SOUL.md define quien es el agente, no que hace (eso lo hacen los skills). Piensa en el SOUL.md como la personalidad + valores + restricciones fundamentales.

Estructura recomendada

# Nombre del Agente — Rol en una linea

Parrafo de identidad: quien eres, que valoras,
como piensas. 2-3 oraciones.

## Philosophical Anchor (opcional pero poderoso)
Una cita o principio que ancla la perspectiva
unica de este agente. Esto evita identity drift.

## Core Directives
Lista numerada de reglas inquebrantables.

## Output Format
Esquema JSON exacto que el agente DEBE producir.
Sin markdown, sin preambulo, solo JSON.

## Constraints
Lo que el agente NO debe hacer nunca.

## Domain Boundaries
Que esta FUERA del alcance de este agente.
Patron descubierto: El Philosophical Anchor es el arma secreta contra identity drift. Un agente con un ancla filosofica clara mantiene coherencia incluso despues de cientos de invocaciones. Sin ancla, el agente tiende a "homogeneizarse" y perder su perspectiva unica.

7 cosas que deben estar en el SOUL.md

#ElementoPor que
1Identidad en 1 parrafoEl agente necesita saber quien es antes de actuar
2Ancla filosofica / perspectiva unicaDiferencia este agente de todos los demas
3Output format (JSON schema)Downstream agents necesitan parsing confiable
4Verdict/decision criteriaCuando aprueba, cuando rechaza, cuando escala
5Severity definitionsminor/major/critical deben significar lo mismo entre agentes
6Domain boundariesPreviene que agentes invadan el territorio de otros
7Constraints (lo que NO hacer)Evita sobre-actuacion y hallucination de problemas
Anti-patron: No pongas instrucciones de skills dentro del SOUL.md. El SOUL define quien eres; los skills definen que haces. Mezclarlos causa confusion cuando los skills evolucionan pero el SOUL no.

4 Skills — Procedimientos Aprendidos

Cada skill tiene frontmatter YAML que Hermes usa para decidir cuando activarlo.

Estructura de un SKILL.md

---
name: nombre-del-skill
description: Una linea que describe que hace
trigger: always | condicion especifica
---

# Titulo del Skill

## Purpose
Que problema resuelve este skill.

## Procedure
Pasos exactos que el agente debe seguir.

## Domain Boundary
Que NO cubre este skill (evita colisiones).

## Known Patterns / Lookup Table
Datos que el skill necesita para funcionar.

## Failure Modes
Que puede salir mal y como manejarlo.

Tipos de trigger

TriggerCuandoEjemplo
alwaysCada invocacion del agenteglossary-enforcement, ab-model-routing
CondicionalSolo cuando se detecta un patron"idiomatic expression detected in source"
CronProgramado (via cron/*.yml)weekly-memory-audit, post-merge-audit

Patrones de skill que funcionan bien

PatronDescripcionEjemplo
GlossaryTabla de terminos canonica que el agente debe respetarglossary-enforcement (translator)
RouterDecisiones de routing basadas en hash o metadataab-model-routing (translator)
DetectorIdentifica patrones especificos y los clasificaambiguity-scoring (quine)
GuardPreviene acciones invalidas (block + log)client_deliverable + test env = BLOCK
Client-SpecificSkill dedicado a un cliente con reglas y glossario propioexample-client (translator)
Leccion aprendida: Si dos skills tienen triggers que se superponen, van a colisionar. Cada skill debe tener un Domain Boundary explicito que diga: "Esto NO lo cubro yo, lo cubre el skill X." Descubrimos 3 colisiones en el proyecto por no hacer esto al principio.

5 Kanban — Orquestacion Multi-Agente

El Kanban de Hermes es el bus de comunicacion entre agentes.

Columnas del Kanban

todo --> ready --> in_progress --> review --> done | blocked --> (revision cycle, max 3 rounds)

Task Decomposition Pattern

Un documento fuente se descompone en una jerarquia de tareas:

[PARENT] translate_document_001
  [CHILD] segment_001 (assignee: translator)
    [CHILD] review_001 (assignee: philosophers)
      parents=[segment_001]  # Dependencia
  [CHILD] segment_002 (assignee: translator)
    [CHILD] review_002 (assignee: philosophers)
      parents=[segment_002]

# Despues de completar todas las reviews:
[PARENT] audit_batch_001
  [CHILD] diff_audit    (assignee: koehn)     parents=[translate_document_001]
  [CHILD] memory_audit  (assignee: cho)       parents=[translate_document_001]
  [CHILD] optimize      (assignee: vaswani)   parents=[translate_document_001]
Patron clave: Usa parents=[...] para crear grafos de dependencias. Un task de review no puede empezar hasta que el task de traduccion termine. Los tasks de auditoria no empiezan hasta que todo el documento este procesado.

Dispatch Intervals — Principio de cadencia

CapaIntervaloRazon
Core (Translator)5sBalance entre responsividad y carga
Philosophers (hot path)3sNecesitan responder rapido al review
Scientists (async)30sNo estan en el hot path, pueden esperar

6 Consensus Engine — Votacion

Multiples agentes votan sobre un resultado con pesos configurables.

consensus:
  strategy: weighted
  quorum: 3              # Los 3 deben votar
  threshold: 2           # 2-of-3 aprueban = pasa
  max_rounds: 3          # Maximo 3 rondas de revision

  weights:               # Pesos por CATEGORIA de issue
    idiom_resolution:
      wittgenstein: 1.5  # Wittgenstein manda en idiomas
      quine: 1.0
      frege: 0.8
    tone_and_style:
      frege: 1.5          # Frege manda en tono
      wittgenstein: 0.8
      quine: 0.8
    ambiguity:
      quine: 1.5          # Quine manda en ambiguedad

  escalation:
    after_max_rounds: "create_task:human_review"
    on_unanimous_block: "create_task:human_review"
Insight: Los pesos deben ser por categoria de issue, no por agente globalmente. Un agente puede ser la autoridad maxima en un tema y tener poco peso en otro. Esto evita que un solo agente domine todas las decisiones.

Escalation chain

Ronda 1: 3 filosofos votan | v (si no hay consenso) Ronda 2: feedback al translator, re-traducir, re-votar | v (si no hay consenso) Ronda 3: ultimo intento | v (si falla) ESCALATION: crear tarea human_review en Kanban

7 Patron Hot Path vs Async

La arquitectura de dos capas es el patron mas potente que descubrimos.

DimensionHot Path (Philosophers)Async (Scientists)
CuandoCada unidad de trabajoPost-merge o cron semanal
Latencia<30s por agenteMinutos, no importa
AccionApprove / Revise / BlockReport + Recommend
Modelo idealRapido, cost-efficientPuede ser mas lento y potente
Scope1 unidad de trabajoBatch-level, cross-agent
Polling3-5 segundos30+ segundos
Patron: Los agentes del hot path son quality gates — bloquean trabajo malo. Los agentes async son optimizadores — mejoran el sistema sin frenar el pipeline. Nunca mezcles estos roles en un solo agente.

Cuando agregar un agente a cada capa

  • Hot path: si el agente debe ver y aprobar CADA unidad de trabajo antes de que avance
  • Async: si el agente analiza tendencias, detecta drift, o optimiza el sistema globalmente
  • Ambos? NO. Divide en dos agentes. Un agente que hace quality gate Y optimizacion va a ser lento en el hot path y superficial en la optimizacion

8 Profile Distributions

Empaquetar agentes como directorios Git instalables.

Los profiles se versionan en Git como directorios con distribution.yaml como manifiesto. Se instalan con:

hermes profile install -y profiles/mi-agente/ --name mi-agente

Que va en Git vs que NO

En Git (version-controlled)Per-machine (gitignored)
SOUL.md, config.yaml, distribution.yamlmemories/, sessions/, state.db
skills/*/SKILL.md + archivos auxiliares.env (API keys)
cron/*.ymllogs/
shared/*.yml (glossary, consensus, budget).hermes/ (runtime completo)
Tip: Configura .gitattributes con Git LFS para archivos *.db. Si alguien accidentalmente commitea un state.db, LFS previene que el repo se llene de binarios.

9 Dual-Provider & A/B Routing

Un solo proyecto, dos proveedores LLM, switch instantaneo.

El patron

# shared/environment.yml
active: production  # test | production

test:
  provider: z-ai
  models:
    translator: glm-4.7-flash       # Gratis
    scientist:  glm-5.1            # Pagado

production:
  provider: anthropic
  models:
    translator: claude-opus-4       # Calidad maxima
    scientist:  claude-sonnet-4    # Cost-efficient

Makefile targets para switch

make env-test    # sed reemplaza model/provider en todos los config.yaml
make env-prod    # lo mismo, pero hacia produccion
make env-status  # muestra ambiente actual
Safety guard critico: El skill de routing debe tener un environment mismatch guard: si el ambiente es test pero la tarea tiene client_deliverable: true, la tarea se BLOQUEA automaticamente. Esto previene que output de modelos baratos llegue a clientes.

A/B entre modelos del MISMO provider

# Hash determinista para asignacion reproducible
cohort = SHA-256(translation_id) mod 2
0 => modelo primario (ej: Opus 4)
1 => modelo alternativo (ej: Sonnet 4)

# Mismo translation_id = siempre mismo modelo
# ~50/50 distribucion sobre N grande
# Sin dependencia de orden de llegada

10 Identity Drift Detection

Los agentes cambian con el tiempo. Hay que detectar cuando pierden su identidad.

El mecanismo

# 1. Generar baseline (una vez)
shasum -a 256 profiles/*/SOUL.md > shared/drift_baseline.json

# 2. El scientist agent recomputa hashes semanalmente
# 3. Si hash actual != baseline => CRITICAL alert
# 4. Cambios deben ser deliberados y documentados

Clasificacion de cambios

TipoDescripcionAccion
COSMETICFormateo, typosActualizar baseline
EVOLUTIONNuevas capacidades dentro de la identidadDocumentar + actualizar baseline
DRIFTContradiccion con identidad originalRevertir (git checkout)
Regla: El SOUL.md baseline siempre debe leerse desde profiles/<agent>/SOUL.md (el distribution directory), NO desde $HERMES_HOME/profiles/ (el runtime). El distribution es la fuente de verdad; el runtime puede estar desactualizado.

11 Skill Collisions & Domain Boundaries

El bug mas comun en multi-agente: dos skills disparando en el mismo input con consejos contradictorios.

Las 3 colisiones que descubrimos

ColisionSkills involucradosSolucion
Idiom vs Pragmatic idiom-localization + pragmatic-context (Wittgenstein) pragmatic-context ahora tiene clausula: "Si idiom-localization ya lo cubre, defer to it"
Ambiguity vs Indeterminacy ambiguity-scoring + indeterminacy-detection (Quine) Cada skill declara: "Este skill NO cubre X, eso lo maneja Y"
Tone vs Sinn/Bedeutung tone-preservation + sinn-bedeutung (Frege) Protocolo de euphemism overlap: tone-preservation tiene prioridad en eufemismos

Patron de Domain Boundary en SKILL.md

## Domain Boundary

This skill covers: [lista de lo que SÍ cubre]

This skill does NOT cover:
- [X] — handled by `skill-name` in the same profile
- [Y] — handled by `other-agent` profile

When both this skill and `overlapping-skill` trigger on
the same input, THIS skill defers to `overlapping-skill`
for [specific aspect] and handles only [its own aspect].
Regla de resolucion: Skill mas especifico gana > version mas alta gana > escalar a humano. Nunca resolver silenciosamente — siempre log.

12 Memory Architecture

Hermes tiene memoria nativa con reflection automatico.

Dos capas de memoria

CapaArchivoQue almacenaIndexado
SemanticaMEMORY.mdPatrones destilados, lecciones aprendidasLegible por humanos
Episodicastate.dbTodos los mensajes, conversaciones completasSQLite FTS5 (full-text search)

Hermes Reflection

Hermes ejecuta pases de reflection automaticos que destilan patrones de state.db hacia MEMORY.md. Tu no escribes MEMORY.md directamente — el agente lo hace.

Problemas de memoria que debes auditar

  • Orphaned entries: Memorias que referencian traducciones que ya no existen
  • Stale patterns: Patrones que no se ven en las ultimas 100 invocaciones
  • Cross-agent contradiction: Agente A aprendio "usar Sie-form" pero agente B aprendio "usar du-form"
  • Over-generic reflections: "Las traducciones deben ser precisas" (inutil)
  • Over-specific reflections: Memorizar una oracion exacta (overfitting)
Patron: Dedica un scientist agent exclusivamente a auditar la memoria de todos los demas. En nuestro proyecto, Cho hace exactamente esto — cross-reference MEMORY.md contra el corpus real, detecta stale entries, y verifica coherencia cross-agent.

13 Token Budget & Cost Control

Cada agente tiene un presupuesto de tokens. Un scientist lo monitorea.

# shared/token-budget.yml
agents:
  translator:
    max_tokens_per_invocation: 4096
    max_invocations_per_hour: 100
    tier: opus      # Mas caro, calidad maxima

  philosopher:
    max_tokens_per_invocation: 2048
    max_invocations_per_hour: 100
    tier: sonnet    # Mas barato, suficiente calidad

adjustment_rules:
  underutilized_threshold: 0.50  # <50% uso = reducir
  saturated_threshold: 0.95     # >95% uso = aumentar
  adjustment_step: 0.15         # +/- 15% por ajuste
Principio de tiering: Solo usa el modelo mas caro para lo que el cliente ve directamente. Todo lo demas (review, audit, optimization) puede ir en un modelo mas barato sin afectar la calidad del deliverable.

14 Logging & Traceability

Cada "run" del pipeline genera una carpeta con el output de cada agente.

corpus/runs/2026-05-30_001/
  manifest.yml                    # Metadata del run
  source/
    segments.json                 # Texto fuente segmentado
  agents/
    translator/output.json        # Traduccion + confidence + flags
    wittgenstein/review.json      # Verdict + critique
    quine/review.json
    frege/review.json
    koehn/audit.json              # Diff audit
    cho/audit.json                # Memory audit
    vaswani/audit.json            # Optimization
  consensus/
    verdict.json                  # Resultado del voto ponderado
  final/
    approved.json                 # Traduccion final aprobada
Patron: Usa un template directory (corpus/runs/.template/) con JSON schemas vacios. Un Makefile target (make new-run) copia el template, auto-genera run_id y timestamp.

15 Git como Mecanismo de Rollback

La inteligencia del agente esta en archivos. Git = rollback instantaneo.

# El agente Wittgenstein aprendio algo malo? Revertir:
git checkout HEAD~1 -- profiles/wittgenstein/
hermes profile install profiles/wittgenstein/

# Un skill causa regresion? Revertir solo el skill:
git checkout HEAD~3 -- profiles/translator/skills/glossary-enforcement/
hermes profile install profiles/translator/

# Comparar la inteligencia de un agente entre dos puntos:
git diff v1.0..v1.1 -- profiles/frege/SOUL.md
git diff v1.0..v1.1 -- profiles/frege/skills/
Principio: Porque SOUL + skills + config son archivos en Git, la "inteligencia" del agente tiene history completo, es diffable, y es revertible. Esto no es posible si la inteligencia vive solo en prompts en codigo o en una base de datos.

16 Makefile Bootstrap Pattern

Un Makefile bien hecho hace que cualquiera pueda levantar el proyecto en 3 comandos.

Targets esenciales

TargetQue hace
make bootstrapCrea .hermes/, instala profiles, muestra next steps
make install-profilesRe-instala los 7 profiles en .hermes/
make test-profilesVerifica que todos los profiles estan OK
make start / stop / statusControl del gateway Kanban
make env-test / env-prodSwitch de ambiente (dual-provider)
make env-statusMuestra ambiente actual
make new-runCrea directorio de run desde template
make cleanBorra .hermes/ (con confirmacion)
Leccion macOS: sed -i '' (con string vacio) es la sintaxis de macOS. En Linux es sed -i (sin argumento extra). Si tu Makefile debe funcionar en ambos, usa una variable SED_INPLACE.

17 Checklist para Nuevo Proyecto Hermes

Copia esta lista cuando arranques un proyecto nuevo.

Fase 1: Estructura

  • Crear directorio con profiles/, shared/, corpus/, docs/
  • Crear Makefile con HERMES_HOME := $(CURDIR)/.hermes
  • Crear .gitignore excluyendo .hermes/, *.db, .env, logs/
  • Crear .gitattributes con LFS para *.db
  • Crear shared/environment.yml con configuracion dual-provider

Fase 2: Profiles

  • Para cada agente: crear directorio en profiles/<name>/
  • Escribir distribution.yaml (manifiesto de instalacion)
  • Escribir config.yaml (modelo, provider, kanban interval)
  • Escribir SOUL.md con: identidad, ancla filosofica, output format, constraints, domain boundaries
  • Decidir: hot path (3-5s poll) vs async (30s+ poll)

Fase 3: Skills

  • Crear skills/<name>/SKILL.md con frontmatter (name, description, trigger)
  • Definir domain boundaries explicitos para evitar colisiones
  • Agregar lookup tables / registries como archivos auxiliares
  • Si hay glossario: crear shared/glossary.yml y skill de enforcement

Fase 4: Orquestacion

  • Definir task decomposition graph (parent/child + dependencies)
  • Configurar consensus engine (strategy, quorum, weights, max_rounds, escalation)
  • Crear shared/consensus-config.yml
  • Crear shared/token-budget.yml con limites por agente

Fase 5: Scientists (Audit Layer)

  • Agente de drift detection: SHA-256 baseline de SOUL.md files
  • Agente de memory audit: cross-reference, stale detection, coherence
  • Agente de optimization: token budget, skill pruning, A/B analysis
  • Configurar cron jobs para audits periodicos

Fase 6: Safety

  • Infinite loop prevention: max_rounds en consensus, task timeout
  • Environment mismatch guard: client_deliverable + test = BLOCK
  • Fallback chain: modelo primario -> secundario -> queue
  • Context window saturation: degradacion progresiva
  • Human escalation: crear tarea human_review automaticamente

Fase 7: Documentacion

  • Wiki en docs/ con pages por tema (overview, architecture, profiles, workflow...)
  • Template de run en corpus/runs/.template/ con JSON schemas
  • Makefile make new-run para crear runs desde template
  • README.md con Quick Start (3-4 comandos para levantar)
